# SoroSense — monorepo

Non-custodial, mobile-first deposit-to-earn app on Stellar with an invisible Sentinel safety engine. An agent auto-allocates each currency bucket to the safest-highest yield; a Sentinel freezes toxic pools. Safety is invisible to the user. pnpm monorepo: `backend/`, `frontend/`, `landing-page/`, `packages/*`, `smart-contract/` (Cargo).

- Run `pnpm install` at the **repo root**, never `npm install` inside a package. One shared lockfile.
- Each track has its own guide — read the `AGENTS.md` / `CLAUDE.md` in the package you're working in (e.g. `frontend/`, `smart-contract/`).
- `docs/brainstorms/` — Product Contracts (WHAT). `docs/plans/` — implementation plans (HOW). Work is tracked in Linear (team `sorosense`); 1 unit = 1 branch = 1 PR, PRs use the `pr-e2e-evidence` template.

## Commands

- `pnpm -r typecheck` / `pnpm -r test` — whole workspace (backend + vault-client use `vitest`).
- `pnpm -C backend test` / `pnpm -C packages/vault-client test` — per package.
- Contract: `cargo test` in `smart-contract/`.
- **Typecheck is a hard gate.** `tsc` is strict with `noUncheckedIndexedAccess` — indexed access is `T | undefined`; prefer `[...arr].reverse()` / guards over `arr[i]` in a way that can leak `undefined`. Tests passing does NOT mean typecheck passes; run both.

## Core invariants (do not violate)

- **Per-currency buckets, never converted** — USD/EUR/MXN funds stay in their own bucket. Blending to USD is display-only (via Reflector FX), never a fund conversion.
- **Safety is invisible** — no risk label/tier/score is ever exposed on a user surface. No `risk`/`label`/`score` field on API results.
- **No chatbot** — the only AI-adjacent user surface is the deterministic `simulate()` projection (math, not an LLM).
- **Read surfaces are read-only** — `backend/src/api/*` and `backend/src/earnings/*` never move funds or write on-chain.
- **`KEEPER_SECRET` is backend-only** — never shipped to the client.

## The DRY seam — `packages/vault-client`

`src/interface.ts` is the single source of the vault's callable shape (`VaultClient`). Backend, frontend, and (via generated bindings) the contract all build against it — never re-declare vault types. `src/mock.ts` (`MockVaultClient`) implements it in-memory for development; generated testnet bindings swap in at integration without any consumer changing an import.

- **Seam reads return `Promise<T>` directly** (e.g. `balanceOf`, `sharePrice`, `assetValueOf`) — NOT `Result`. `Result` lives in `backend/src/lib/result.ts`; `packages/vault-client` must not depend on `backend`.
- Amounts/shares/prices are `bigint`. Share price is fixed-point `PriceRay` scaled by `SHARE_PRICE_SCALE`; base (no yield) == the scale. The mock mirrors the contract's virtual-offset NAV math and exposes a **test-only** `simulateYield(currency, amount)` to raise NAV without minting shares.
- The contract backs those two reads with the public views `share_price(currency)` / `value_of(user, currency)` (`smart-contract/contracts/vault/src/lib.rs`); its `SHARE_PRICE_SCALE` in `shares.rs` must stay equal to the seam's. **Yield does not accrue on-chain yet** — `total_assets` moves only on deposit/withdraw — so the live `share_price` reads exactly the scale until mark-to-market NAV ships. Only the mock/test hooks can lift it.
- **Auto-compound is a per-depositor economic preference, separate from consent** (STE-38/STE-40). `setAutoCompound(depositor, enabled)` (depositor-signed write) + `autoCompoundEnabled(depositor)` (read, **default true** — unset = ON). Toggling it never touches `setPolicyConsent`/`hasConsent` (the safety mandate, KTD3). Revoke = stop reinvest only; allocate/rebalance/freeze-exit are unaffected.
- **Two client impls, config-selected (STE-21).** `mock.ts` (`MockVaultClient`) is the default everywhere — dev + the whole test suite run against it, no network. `real.ts` (`RealVaultClient`) wraps the generated testnet bindings in `packages/vault-client/bindings/` (regenerate via `stellar contract bindings typescript --network testnet --contract-id <id>`; src is committed). Reads → `Promise<T>`; writes → a `PreparedTx` that assembles+signs+submits. **Pool registry:** writes/`poolStatus` take the seam's `PoolId` slug but the contract wants an `Address` — `RealVaultClient` resolves it through an **injected** `resolvePool(slug)→Address` (the backend builds it from env; no pool address is hardcoded in the seam).

## Backend conventions

- Remote/network reads return the typed `Result<T>` (`backend/src/lib/result.ts`) — never throw; missing/stale reads are fail-closed in the Sentinel.
- Deterministic, injectable dependencies: pass a `clock: () => number` instead of calling `Date.now()` in module cores; pass event/data sources in rather than reading chain directly (real readers are deferred to integration).
- `backend/src/earnings/` — earnings-history capability: `cost-basis.ts` (reconstruct net contributions per (user,currency) from `Deposit`/`Withdraw` events), `snapshotter.ts` (record the global per-currency share-price time series + Day/Week/Month/Year bucketing). `backend/src/api/earnings.ts` (`getEarnings`) composes those with the vault seam and injected FX into the read-only, blended-USD deposited-Earn view (balance, value-weighted APY, total earned, value+earned timeline, per-month breakdown). Earned = native yield (`value − contributions`) summed to USD — FX movement is never earnings; a failed FX read returns a typed `Result` error, never a silent $0. Per-user earned attribution (the value×price − contributions timeline) lives here, not in the snapshotter. **Timeline shape:** each `ChartPoint` carries **both** `valueUsd` and `earnedUsd`, from ONE replay (`stateAt(t)`), sampled at the **union of snapshot timestamps and the user's event timestamps** — so a deposit steps the value chart before the next snapshot tick, and a freshly-booted server still has a chart. Each `BucketBreakdown` carries its own `earnedUsd` (summing to the headline). Because the contract does not accrue yet (`share_price` == the scale), in real mode `earnedUsd` is honestly **0** and `valueUsd` is a **step function** on real deposits/withdrawals — never fabricate growth. `backend/src/http/openapi.ts` documents these fields field-by-field (`openapi.integration.test.ts` forbids drift).
- `backend/src/mastra/allocator.ts` — `classifyBucket` stays **pure and per-currency-bucket** (no per-depositor branch); the auto-compound preference is honored at the **effect layer**. `runAllocatorTick` optionally takes `depositors` + `autoCompoundEnabled` (the seam read); `gateCompound` fires `effects.compound(currency, pool, depositor)` only for ON depositors and is **fail-closed** — an unreadable preference is treated as OFF (never reinvest unverified). Omit both fields → legacy ungated pool-level compound. Rebalance/freeze-exit never pass through the gate.
- `backend/src/http/` — the **HTTP surface** (STE-21 Fase B): a thin Hono app (`app.ts`, listen-free `createApp` — `server.ts` is the only place that binds a port) exposing the composed reads (`getHoldings`/`getActivity`/`getEarnings`/`getFundingOptions`/`getRates`/`getPool`) over GET routes, transport-only (re-implements no read logic). `bigint`→decimal string at the boundary via one shared replacer; a failed `Result` → shaped non-200, never a silent 200; read-only (no route reaches a write). **Faucet** (`faucet.ts` + `faucet-minter.ts`): env-gated `POST /faucet` mints self-issued testnet USDC/EURC (STE-46) — **`FAUCET_ISSUER_SECRET` is backend-only, reaches only the real minter, never a response**; USD/EUR only; per-address rate-limit; no-trustline → 409 `changeTrust` hint. Mounted only when the faucet env is present (inert on mainnet).
- `backend/src/chain/event-reader.ts` — the **on-chain event reader** (STE-21 Fase C): `readVaultEvents(EventSource)` decodes the vault's events into `VaultEvent` (cost-basis) + `UserActionEvent` (activity), so history is sourced from chain instead of injected fixtures. Injectable `EventSource` (real `makeRpcEventSource` wraps `rpc.getEvents`; tests feed canned `ScVal` pages through the real pure derivations). **Event topics are snake_case** (`deposit`/`withdraw`/`consent_set`/`auto_compound_set`/`exit_approved`); `Currency` decodes as `['Usd']`. `UserActionEvent` includes `auto-compound` (from `auto_compound_set`). Note: `exit_approved` carries no depositor on-chain, so `approve-exit` isn't attributable from the event alone (documented follow-up — never fabricated).
- `backend/src/chain/` + `backend/src/http/realtime.ts` — the **realtime loops** (U1): Stellar RPC has **no event streaming** and retains events only ~7 days, so realtime is a poll, and the poll must **accumulate** rather than re-drain from the deploy ledger (KTD1). `event-store.ts` (`InMemoryEventStore`) accumulates raw events deduped by RPC event `id`, ledger-ordered, carrying the cursor; `poller.ts` (`makeChainPoller`) polls from that cursor → ingests → **re-decodes the WHOLE accumulated set** via `decodeEvents(ordered)` (so `seq` is monotonic across polls — a per-page decode would restart it at 0 and scramble the feed) → `onUpdate`. **The poller is fail-soft: `poll()` never rejects**, because it runs under `startScheduler` and a rejected handler would poison the interval; the holder keeps the last good decode. `startRealtime(deps)` is the **live-only** wiring (returns `null` under `isIntegrationEnv()===false` having constructed nothing — no source, no `rpc.Server`, no scheduler, so offline stays zero-network): resolve `startLedger` (`VAULT_START_LEDGER`, else latest−~120k ledgers, clamp ≥1), one immediate poll + snapshot at boot, then `EVENT_POLL_MS`/`SNAPSHOT_INTERVAL_MS` loops. **Refresh is field-reassignment on the deps holder** (`deps.earnings.events` / `deps.activity.userEvents`) — routes dereference them per request, so `app.ts` never changes (KTD2). It lives beside `server.ts` because `server.ts` binds a port on import and cannot be imported by a test. History is in-process: a restart loses it (durable store = follow-up).
- `backend/src/tools/price.ts` — **FX is a real on-chain oracle read** (U1c), never a REST call. Reflector is a Soroban **SEP-40 oracle contract**; the `https://reflector.stellar.org/price/{asset}` endpoint it replaced never existed (the host does not resolve), which is what made every live `/earnings` a 503. `makeReflectorReader` simulates `lastprice(Other(<symbol>))` on `REFLECTOR_ORACLE_ID` (testnet default `CCYOZJCO…RN63`) over `STELLAR_RPC_URL` and decodes `Option<PriceData{price:i128,timestamp:u64}>`. **The scale comes from the feed, not a constant:** with `REFLECTOR_DECIMALS` unset it reads the oracle's own `decimals()` (= **14**; `1.1443 USD/EURC` = `114433043263595`) once per reader — a pinned scale against a repointed oracle would multiply every rate by a power of ten, and a 10,000× rate is not a crash, it's a €100 bucket shown as $1.1M. On this feed `base()` is **USD**, so a price IS that asset's USD rate. Injectable `OracleSource` like `event-reader.ts` (real one wraps `rpc.simulateTransaction`, built ONCE per reader, 8s timeout — the SDK's RPC client has none; tests feed canned `ScVal`s through the real decode) — **the suite never touches the network**; the mock/dev server still uses the deterministic stub FX. Everything fails closed as a typed `Result`, **including construction** (`new rpc.Server`/`new Contract` throw on a bad URL or a typo'd contract id — the very knobs we advertise as `.env`-editable, so a typo must be a 503, not an unshaped 500): unconfigured/misconfigured/RPC failure → `unavailable`, a stall → `timeout`, `None` → `not_found`, malformed **or zero** price → `parse` (a rate of 0 would blend real money to a silent $0). **Symbols are config (KTD6):** this feed lists TOKEN symbols (`assets()`: USDC, EURC, XLM…), so EUR is priced as **`EURC`** — there is no `EURUSD` pair on it and **no MXN at all**, so `FX_SYMBOL_MXN` defaults to none and an unpriceable bucket fails closed rather than being blended 1:1 into dollars. USD is the numéraire (rate 1, no read). Consequently `getEarnings` requires a rate only for buckets that **hold value or have history** — an untouched bucket contributes `0 × rate = 0` exactly, so it can't 503 the view (same posture as `getHoldings`, which already prices only the buckets it shows).
- **Yield does not accrue on-chain yet, so in live mode `earned` is exactly 0** — that is the truth, not a bug. Never fabricate growth to cover it: an empty history source is what made `/earnings` report a user's whole principal as profit (`earned = value − 0`), and the realtime test pins it dead.
- `backend/src/keeper/` — the **keeper runner** (STE-21 Fase D): `makeKeeperRunner` builds real `AllocatorEffects` over the `RealVaultClient` keeper writes (keeper-signed), driving the deterministic `runAllocatorTick`. Operator actions `freezePool`/`unfreezePool`/`runTick` + a thin CLI (`cli.ts`: `freeze|unfreeze|tick <ccy>`) — **manually invoked** for the demo (no autonomous loop). Refuses real writes in mock mode. `classifyBucket` stays pure — the runner is the effect layer only. The **pool registry** (`tools/vault.ts` `demoPoolFor`/`buildPoolRegistry`) is the ONE place a currency maps to its demo pool slug + `BLEND_POOL_<CCY>` address; threaded into `RealVaultClient` as `resolvePool`.
- **API QA for the HTTP surface** — `backend/src/http/openapi.ts` is a hand-authored OpenAPI 3.1 spec served at `/openapi.json` + Swagger UI at `/docs`, kept honest by `openapi.integration.test.ts` (every registered route must be documented — the spec can't silently drift). We use **Bruno** (not Postman): `bruno/sorosense-api/`. **Schemathesis** fuzzes the spec (`qa/schemathesis/run.sh`, via `uvx`) and **K6** load-tests the reads (`qa/k6/`) — both against a **mock-mode** server (offline: `server.ts` uses a deterministic stub FX when the integration env is absent, so the mock server never hits the network). Schemathesis/K6 are operator-run (not CI gates); evidence in `docs/tests/api-qa/`. No DB/ORM/Testcontainers here — state is on-chain + in-memory, auth is the wallet.
- `backend/src/api/activity-feed.ts` (`getActivity`) — read-only source-of-truth for the Activity **All / Yours / Automated** tabs, so the frontend "Yours" filter stops depending on a fixture. Merges the agent feed (`backend/src/api/activity.ts` `ActivityLog`, now carrying `actor: 'you' | 'agent'`, default `'agent'`) with user actions derived from an injected `UserActionEvent` stream (`backend/src/api/user-activity.ts` `deriveUserActivity`, pure/deterministic like `cost-basis.ts`, real reader deferred to U20). `actor` maps to the tabs (unset=All, `'you'`=Yours, `'agent'`=Automated); `depositor` narrows only the user rows (agent rows are pool-level). Deterministic by monotonic seq, read-only, no risk/label/score field. Covers deposit/withdraw/sign-mandate/approve-exit.
- `backend/src/api/holdings.ts` (`getHoldings`) + `backend/src/api/funding.ts` (`getFundingOptions`) — read-only source-of-truth for the Home bucket rows and Add-funds list, so the frontend stops hardcoding venue/APY (`frontend/lib/vault/data.ts`). `getHoldings` is a drop-in superset of the frontend `BucketView`; both derive venue/kind/APY from `backend/src/api/venue-meta.ts` (`resolveVenue`/`bestSafeVenue`/`kindLabel`, catalog-derived, the DRY seam between the two). RWA funding options omit APY; no risk/label/score field anywhere.
- `backend/src/api/rates.ts` (`getRates`) + `backend/src/api/pools.ts` (`getPool`) — the last two display fixtures, moved behind the seam (U4). **`GET /rates`** answers what `/holdings` structurally cannot: `getHoldings` omits a zero-share bucket (correctly — there is no holding to report), so the **unfunded** bucket's rate (the Earn empty-state hero, the simulator) had nowhere to come from but a frontend constant. It is user-independent (no depositor) and quotes `bestSafeVenue` — the venue the agent *would* allocate to — so the rate a user is promised before depositing is the rate the keeper actually chases. A currency with no vetted venue is **omitted**, never quoted at 0%. **`GET /pools/:id`** names + rates one vetted pool: the exit-approval sheet's target, which the frontend's `POOL_META` could only guess at (on-chain, `ExitProposal.toPool` is a catalog slug that map has never carried). An unknown **or trap** id → shaped **404**, never a 200 carrying `null` — a blank sheet asks the user to approve a move to a pool with no name. Both go through the **same `venue-meta.ts` seam** (there is no second catalog): pure, no vault, no FX, no network, no risk/label/score field. Documented in `openapi.ts`; the drift test normalizes Hono's `:id` to the spec's `{id}` so a parameterized route is still checked, not exempted.
